# SmartAnalysis 开发讨论过程

## 第一阶段：设计讨论（凌晨）

### 议题 1：数据分析流程全面重构
- 废弃原有静态图表 Tab，改为"提出问题 → 上传文件 → AI 智能分析"
- Tab 结构调整为五个：数据接入 → 数据理解 → 智能分析 → 模型训练 → 预测沙盘

### 议题 2：AI 角色定位
- AI 只做推荐/规划 + 解读，不生成代码
- Python 执行预定义分析函数，双向通道结构，JSON 协议通信
- 分析体量从 8 个扩展到 24 个，注册表模式管理

### 议题 3：UI 设计方向
- 智能分析采用"分析笔记本"模式：左侧历史时间线 + 右侧分析卡片
- 废弃"AI 自动分析，不等用户确认"的 B 方案
- LLM 配置从右下角弹出改为 Tab 1 侧边栏直达
- 设计规范：去掉 emoji，SVG 图标，统一浅色系，card 形式

## 第二阶段：开发实现

### 议题 4：合并 LLM 调用优化
- 翻译和规划原本两次独立 LLM 调用（6-10s），合并为一次 combined 调用（~3s）
- 在 `_call_llm_stream` 中实现 token 转发，前端从第 1 秒就能看到输出

### 议题 5：分析函数组织
- 四层 24 个函数：分布层 5 + 关系层 7 + 趋势层 6 + 查询层 4 + 质量层 2
- 每个函数自带默认图表类型，AI 可覆盖
- `generate_ai_functions_catalog()` 自动生成注册表并注入 LLM Prompt

### 议题 6：回归诊断增强
- 训练管道增加残差 QQ 图、预测 vs 实际散点图、VIF 多重共线性、K-fold CV、Ridge/Lasso 对比
- 伪代码升级为真实诊断分析

### 议题 7：沙盘可视化重构
- 从单一彩色仪表盘 → 改为贡献瀑布图 + 场景对比柱状图 + 敏感性龙卷风图
- 新增预测区间、场景保存、自动寻优

## 第三阶段：用户验证

### 议题 8：认证方案选择
- SQLite 而非 MySQL，零安装，一个文件，项目级可移植
- 自实现 JWT 而非 pyjwt，用 hashlib + base64，零外部依赖
- PBKDF2-SHA256 密码哈希，100k 轮迭代

### 议题 9：登录态管理
- sessionStorage 存 token，全局 fetch 拦截器自动附加 Authorization 头
- 401 自动退出 + 提示
- 登录后先弹出隐私声明，可选"下次不再提醒"（localStorage）

### 议题 10：历史记录
- 登录后展示历史面板，"继续上次"一键恢复会话
- 同一文件连续分析自动覆盖旧记录（去重）
- 每用户限制 50 条上传 + 100 条分析
- Admin 用户在管理页面可查看全局记录 + 逐条删除

| 问题 | 原因 | 修复 |

## 第六阶段：AI 分析鲁棒性增强（2026-05-26）

### 问题发现

用户提问"销售额（Quantity × UnitPrice）与哪些因素有关"，LLM 正确识别了计算列 `Revenue = Quantity * UnitPrice`，但因列映射与 actions 之间命名不一致，`describe_distribution` 和 `compare_groups` 报 KeyError `'Revenue'` 列不存在。

### 根因分析

断裂点有三：
1. **命名不一致**：LLM 在 column_mappings 声明 `{name: "销售额", formula: "Quantity * UnitPrice"}` 但在 actions 中引用 `{column: "Revenue"}`——名字对不上
2. **computed 列未声明**：LLM 可能直接跳过 column_mappings 中的 computed 声明，只在 actions 中引用计算列名
3. **静默失败**：`apply_column_mappings` 异常被 `try/except` 吞掉，`analysis_df` 退化回原始 `df`，分析照常执行但全部报错

### 解决方案：三层防御

**Prompt 层**（`llm_agent.py`）：
- 重写 `_format_columns_desc`：每列附带类型标签、取值范围/均值、高频值 Top 5、前 3 行样本值，让 LLM 先推断列的业务含义再匹配用户意图
- 强化 `_COMBINED_PROMPT`：新增 6 条硬性约束，包括 actions 与 column_mappings 列名一致性、computed 列必须声明、无法分析时诚实说明
- 更新 `_INTERPRETATION_PROMPT`：增加错误处理规则，引导 LLM 解释列缺失原因并给出修正建议

**校验层**（`main.py`）：
- 执行前通过 registry 的 `input_schema` 自动识别哪些 param 是列引用（描述中含"列"字），与 DataFrame 实际列名比对
- 不存在的列立即返回 SSE error event，含"当前可用列: [...]"，阻止 KeyError 传播
- 非列参数（method、operator、segment_condition 等）自动跳过

**解读层**（`main.py` + `llm_agent.py`）：
- `mapping_logs` 和 `_missing_col_info` 注入 LLM 解读上下文
- LLM 可精确解释哪些列缺失、可能原因及修正方向

### 效果

- 列名不匹配时，用户收到"列 'Revenue' 在数据中不存在。当前可用列: [Quantity, UnitPrice, Country, ...]"的明确提示
- LLM 在解读阶段可给出"请确认计算销售额的公式是否正确，或检查数据中是否有已有的销售额字段"等建设性建议
- 无法分析时不再硬编，而是诚实说明"根据上传数据，无法完成本次分析，因为……"

### 延伸改进

基于同样的"LLM 不稳定 + 用户体验差"思路，连带了以下优化：

- **LLM JSON 解析多级回退**：在 `_extract_json` 中增加去除自由文本、尾逗号修复、单引号转双引号三重回退
- **沙盘滑块范围收紧**：从全局 min/max 改为 P1/P99 分位数，避免极端异常值（如 Quantity 的 -80995）撑大滑块
- **训练前数据预览**：新增 `/preview-train` 端点，训练前展示有效样本数和 NaN 丢弃率
- **非数值特征拦截**：选了 Country 等文本列作特征时，后端明确返回"不支持用于回归分析"并给出具体列名
- **高基数特征检测**：自动识别 CustomerID 等唯一值占比 >50% 的特征，标记为"可能是 ID 列"
- **统一错误弹窗**：所有用户可见的错误（上传失败、训练失败、SSE 异常、401 过期）统一改为居中模态弹窗
- **上传 50MB 限制**：前端 + 后端双重校验
- **SSE 异常事件**：LLM 调用、列映射、解读三阶段失败均发送 `event: error`
- **会话上下文 LRU**：`_session_context` 限制最多 20 个文件，防止内存无限增长
- **技术文档清理**：移除技术讲解.md 中所有过期的行号引用
