## 1. 总体架构

### 1.1 技术分层

```
浏览器 (前端)
 │
 ├─ HTTP/JSON ──→ FastAPI (:8000) ──→ services/ ──→ pandas/scikit-learn
 │ │
 └─ SSE (流式) ──→ StreamingResponse ──→ LLM Agent (httpx async)
```

- **前端**：纯静态 HTML/CSS/JS，零框架依赖，ECharts 5.5 渲染图表
- **后端**：Python FastAPI，11 个 REST 端点 + 1 个 SSE 流式端点
- **存储**：SQLite (用户数据) + 文件系统 (上传文件)
- **AI**：OpenAI-compatible API，自实现 JWT 认证

### 1.2 项目结构

```
project/
├── frontend/
│ ├── index.html # 5 个 Tab + 模态框 + 侧边栏
│ ├── app.js # 全前端逻辑 (~1900 行)
│ └── style.css # 设计系统 (~1200 行)
├── backend/
│ ├── main.py # FastAPI 入口，11 个路由
│ └── services/
│ ├── auth.py # 用户认证 (SQLite + JWT)
│ ├── data_engine.py # 文件读取 + Schema 提取
│ ├── data_profiler.py # 数据画像
│ ├── data_cleaner.py # 数据清洗
│ ├── analysis_registry.py # 分析函数注册表 (24 个)
│ ├── analysis_functions.py # 分析函数实现
│ ├── llm_agent.py # LLM 代理 (流式 + 规划)
│ └── ml_trainer.py # 线性回归训练
└── requirements.txt
```

### 1.3 前后端通信

前端通过 `fetch()` 调用后端 API，关键设计：

- **同源开发**：前端 `python -m http.server 3000`，后端 `uvicorn :8000`，CORS 全开
- **全局 fetch 拦截**：`app.js` 重写了 `window.fetch`，自动附加 `Authorization` 头和 401 处理
- **SSE 流式**：`POST /analyze` 返回 `text/event-stream`，前端 `ReadableStream` 逐块解析

## 2. 后端核心服务

### 2.1 数据引擎 (`services/data_engine.py`)

**文件读取与编码容错**

```python
_CSV_ENCODINGS = ["utf-8", "utf-8-sig", "gbk", "gb18030", "latin1", "iso-8859-1"]
```

采用**逐级回退**策略读取 CSV：从 UTF-8 开始，失败则尝试下一种编码。GBK/GB18030 处理中文乱码，Latin-1 作为最终兜底（不会丢字节）。这是处理真实世界"脏数据"的关键防线。

**类型推断** (`data_engine.py`)

对每列独立推断类型，优先级：
1. Boolean（二分值 + 布尔字面量匹配）
2. Numeric（Pandas 原生数值类型检测）
3. Datetime（已转换类型 或 object 列尝试 `pd.to_datetime` 成功）
4. Categorical（低基数字符串，nunique ≤ 50 或 比率 < 20%）
5. Text（其余字符串）

**性能优化**

- ≥50,000 行自动采样 (`_MAX_ANALYSIS_SAMPLE = 50_000`)
- Excel 上传后自动生成 CSV 缓存 (`_ensure_csv_cache`)，38s → 2s

### 2.2 数据画像 (`services/data_profiler.py`)

对每列生成：
- **分布统计**：min, Q1, median, Q3, max, mean, std
- **Sparkline 迷你图**：20-bin 直方图数据压缩
- **异常值检测**：IQR 方法 (Q1-1.5IQR, Q3+1.5IQR)
- **数据质量问题标记**：高缺失 (>30%)、常量列、类型建议（文本存日期）

### 2.3 数据清洗 (`services/data_cleaner.py`)

支持的操作类型：

| 操作 | 实现 | 参数 |
|------|------|------|
| 缺失值填充 | `pd.fillna()` | mean/median/mode/forward/backward/custom |
| 异常值处理 | IQR 截断或剔除 | cap/remove/keep |
| 重复行删除 | `df.drop_duplicates()` | keep=first/last |
| 常量列删除 | `nunique ≤ 1` 检测 | 自动 |
| 列删除 | `df.drop(columns=...)` | 指定列名列表 |

清洗后另存为 `cleaned_原文件名_时间戳.csv`，原始文件不受影响。

### 2.4 回归训练 (`services/ml_trainer.py`)

**五阶段管道**：

```
Stage 0: 输入解包 + 参数校验
Stage 1: 列存在性检查
Stage 2: 强制数值化 (pd.to_numeric, errors="coerce")
Stage 3: Y 缺失删行 + X 中位数填补 (SimpleImputer)
Stage 4: 拟合 LinearRegression
Stage 5: 诊断计算
```

**多维诊断**：

| 诊断 | 方法 | 说明 |
|------|------|------|
| VIF | `_compute_vif()` | 方差膨胀因子，>5 警告，>10 严重 |
| 残差分析 | `_residual_analysis()` | QQ 图数据 + 预测vs实际 + Durbin-Watson |
| K-fold CV | `_cross_val_r2()` | 5-fold 交叉验证 R² |
| 模型对比 | `_compare_models()` | Linear vs Ridge vs Lasso (80/20 split) |

**标准化特征重要度**：

```
β_std = β_raw × (σ_x / σ_y)
```

在未标准化数据上训练，事后计算 β_std。好处：保留原始系数用于沙盘预测，同时提供标准化系数用于跨特征重要性比较。

**非数值特征拦截**（2026-05-26 新增，Stage 2 后）：

`pd.to_numeric` 后检测每个特征列是否为全 NaN（选了文本列如 Country 时发生），若是则立即 `ValueError`："以下特征列无法转换为数值，不支持用于回归分析"——而非让报错传播到 `SimpleImputer` 或 `LinearRegression.fit()` 产生难以理解的系统错误。

**高基数特征检测**（2026-05-26 新增）：

训练前检查每个特征的 `nunique / n_samples`，若 > 0.5 则标记为"可能是 ID 列"（如 CustomerID 有 4372 个唯一值但只有 5000 行样本），在训练结果 `warnings` 字段中返回。

**沙盘范围收紧**（2026-05-26 修改，Stage 5）：

`feature_stats` 的 `min`/`max` 从全局极小/极大值改为 P1/P99 分位数，避免极端异常值（如 Quantity 的 -80995）撑大滑块范围导致预测脱离实际。

## 3. AI 智能分析系统

### 3.1 分析函数注册表 (`services/analysis_registry.py`)

24 个预定义函数，按层次组织：

| 层次 | 函数 | 说明 |
|------|------|------|
| 分布 | `describe_distribution`, `describe_frequency`, `detect_outliers`, `test_normality`, `analyze_composition` | 数值画像、分类频次、异常检测、正态检验、构成占比 |
| 关系 | `correlate`, `cross_tabulate`, `compare_groups`, `compare_pairs`, `correlation_matrix`, `dimension_reduce`, `cluster_analysis` | 相关性、交叉分析、ANOVA、Tukey HSD、PCA、K-Means |
| 趋势 | `timeseries_line`, `timeseries_decompose`, `timeseries_growth`, `moving_average`, `rank_top_n`, `pareto_analysis` | 时序分析、趋势分解、增长率、排名、帕累托 |
| 查询 | `data_lookup`, `anomaly_explain`, `filter_aggregate`, `segment_profile` | 精确查询、异常解释、条件过滤、分段画像 |
| 质量 | `profile_missing`, `profile_duplicates` | 缺失值画像、重复分析 |

每个函数返回标准格式 `{chart_data, stats, table, chart_type}`，所有值 JSON 安全。

**注册表自动同步 Prompt**：

`generate_ai_functions_catalog()` 遍历注册表生成 Markdown 格式的函数目录，直接嵌入 LLM system prompt。加新函数只需三步：写 Python → 注册 → Prompt 自动更新。

### 3.2 分析函数实现 (`services/analysis_functions.py`)

核心设计原则：
- 采样保护：`_sample()` 对大数据集限制 50k 行
- 纯函数：不修改输入 DataFrame，返回新副本
- JSON 安全：`_safe()` 转换 NumPy → 原生 Python，NaN → null

**关键函数技术细节**：

`correlate`：Pearson + Spearman 双方法，输出散点图点阵 + 趋势线 + r/p 值。趋势线用 `np.polyfit(x, y, 1)` 计算。

`timeseries_decompose`：移动平均去趋势 → 去趋势序列求周期均值 → 季节分量 → 残差 = 原序列 - 趋势 - 季节。季节性强度 = σ²_seasonal / (σ²_seasonal + σ²_residual)。

`cluster_analysis`：K-Means + 自动 k 选择（遍历 2-8，选最高轮廓系数）。PCA 降维到 2D 可视化。

### 3.3 LLM 代理 (`services/llm_agent.py`)

**三层 Prompt 架构**：

```
翻译层 (_COMBINED_PROMPT, )
 ├── 模糊词翻译规则 (8 条)
 ├── 列映射 (direct / computed)
 └── 函数目录嵌入 (从 registry 自动生成)

规划层 (同上，合并后单次调用)
 ├── 24 个函数 → 选择 1-4 个
 ├── 图表类型匹配
 └── 输出结构化 JSON

解读层 (_INTERPRETATION_PROMPT, )
 ├── 分析结果 → 自然语言
 ├── 追问建议
 └── 3-5 段，每段 2-4 句
```

**翻译+规划合并**：

将翻译和规划两次 LLM 调用合并为一次，减少网络往返。2026-05-26 强化后，Prompt 分为三步：

**第 1 步 — 理解数据**：引导 LLM 阅读每列的丰富描述（类型标签、取值范围、均值、高频值、样本值），主动推断列的业务含义（如"Quantity 范围 1~80995 均值 9.55 → 很可能是销售数量"），而非仅做关键字匹配。

**第 2 步 — 概念翻译**：将用户问题中的业务概念映射到实际列名。direct 类型直接引用原始列，computed 类型给出 pandas eval 表达式。关键约束：column_mappings 必须包含 actions 中将引用的每一个列名。

**第 3 步 — 分析规划**：从 24 个函数中选择 1~4 个。如果确实无法匹配，actions 留空，在 plan_text 中诚实说明原因。

输出单个 JSON：
```json
{
 "translation": "对问题的操作性翻译",
 "column_mappings": [{"concept":"销售额", "type":"computed", "formula":"UnitPrice*Quantity"}],
 "plan_text": "分析思路 或 根据上传数据，无法……因为……",
 "actions": [{"function":"timeseries_line", "params":{...}, "chart":"line"}]
}
```

**6 条硬性约束**（违反将导致分析失败）：
1. **列名一致性**：actions.params 中的每个列名必须先出现在 column_mappings 的 `name` 字段中
2. **computed 列必须声明**：公式计算列需先在 column_mappings 中声明 type="computed"
3. **formula 只能引用原始列名**：防止 computed 列之间嵌套引用
4. **无法分析时不要硬编**：actions 置空，plan_text 诚实说明
5. **优先尝试但不勉强**：匹配不确定时选择最有把握的分析
6. **直接输出 JSON**：不要用代码块包裹

**列描述增强** (`_format_columns_desc`, )：

为 LLM 构造远丰富于旧版 `列名 (dtype) [范围]` 的列描述。每列包含：
- 中文类型标签（数值/分类/日期/文本/布尔）
- 数值列：取值范围 + 均值
- 分类列：唯一值数 + 高频值 Top 5
- 全部列：前 3 行样本值

效果：LLM 看到 `Quantity (数值) 范围: -80995~80995 均值: 9.55 样本: [6, 8]` 可自行推断"这是销售数量"；看到 `Country (分类) 38 个唯一值 高频: [United Kingdom, Germany, France]` 可自行推断"这是国家"。

**解读层增强** (`_INTERPRETATION_PROMPT`, )：

新增错误处理规则：若结果中含 `[列映射日志]` 或 `[列缺失提示]`，LLM 需解释具体哪些列缺失及修正建议；全部失败时用"根据上传数据，无法完成本次分析，因为……"诚实说明。

**流式调用** (`_call_llm_stream`, )：

使用 `httpx.AsyncClient.stream()` + SSE token 转发。参数 `stream: true` 使 LLM 逐 token 返回，后端通过 `async for line in resp.aiter_lines()` 实时推送给前端。

**双模切换**：

- `actions` 非空 → 分析管道（翻译→校验→执行→解读→画图）
- `actions` 为空 → 对话回退（调用 `ask_data` 直接回答）

### 3.4 SSE 事件流 (`main.py`)

```
POST /analyze
 ↓ SSE events
event: translation → 概念翻译 + 列映射
event: plan → 分析计划
event: status → 状态更新（分析中/解读中）
event: progress → 执行进度 (step/total)
event: result → 单步分析结果 (chart_data + stats + table)
event: interpretation → AI 解读 (逐 token 流式)
event: done → 完成
event: error → 错误信息
```

**执行前列名校验**（2026-05-26 新增，）：

在 `apply_column_mappings` 之后、每个 action 执行之前，增加了一层列名守卫：

1. 通过 registry 的 `input_schema` 识别哪些 param 是列引用（描述中含"列"字的参数，如 `column`、`value_col`、`group_col`、`columns` 等）
2. 将列参数的值与 `analysis_df.columns` 比对
3. 不存在的列 → 立即返回 SSE `result` error event，包含"当前可用列: [...]"
4. 非列参数（`method`、`operator`、`segment_condition` 等）自动跳过，不会误判

这层守卫阻止了 KeyError 传播到 LLM 解读阶段——此前 LLM 收到 KeyError 后会猜测原因，往往给出误导性建议。现在错误信息精确包含缺失列名和可用列列表，LLM 在解读阶段可以给出准确的修正建议。

**映射诊断注入**：`mapping_logs` 和 `_missing_col_info` 在解读阶段注入 analysis_results 摘要，让 LLM 能解释列映射的成败过程。

**JSON 安全保护**：

`_SafeEncoder` 继承 `json.JSONEncoder`，递归遍历所有值：
- `NaN/Inf` → `null`
- `np.integer/np.floating` → Python int/float
- `np.ndarray` → list
- `pd.Timestamp` → ISO 字符串

### 3.5 多轮对话上下文 (`main.py`)

```python
_session_context: dict[str, str] = {} # key = filename, value = 历史摘要
```

每次分析完成后，自动保存"提问 + 分析结果摘要"到 `_session_context`。下次提问时，上下文通过 prompt 的 `{previous_context}` 传入，LLM 能看到上轮分析的具体数值。

对话回退路径同样带上下文：将历史结果拼入提问文本，让 `ask_data` 基于已知数据回答。

## 4. 用户认证系统

### 4.1 数据存储 (`services/auth.py`)

**SQLite 数据库** (`backend/auth.db`)，三张表：

```sql
CREATE TABLE users (
 id INTEGER PRIMARY KEY AUTOINCREMENT,
 username TEXT UNIQUE NOT NULL,
 password_hash TEXT NOT NULL,
 salt TEXT NOT NULL,
 role TEXT NOT NULL DEFAULT 'user',
 created_at TEXT NOT NULL
);

CREATE TABLE upload_history (
 id INTEGER PRIMARY KEY AUTOINCREMENT,
 username TEXT NOT NULL,
 filename TEXT NOT NULL,
 original_name TEXT NOT NULL,
 uploaded_at TEXT NOT NULL
);

CREATE TABLE analysis_history (
 id INTEGER PRIMARY KEY AUTOINCREMENT,
 username TEXT NOT NULL,
 filename TEXT NOT NULL DEFAULT '',
 question TEXT NOT NULL,
 result_json TEXT,
 created_at TEXT NOT NULL
);
```

**设计要点**：
- 无外部数据库依赖，`auth.db` 随项目文件走
- 首次启动 `_init_db()` 自动建表并插入内置账号
- 历史记录自动限制条数（上传 50 条/用户，分析 100 条/用户）

### 4.2 密码安全 (`auth.py`)

```
PBKDF2-SHA256(密码, 随机盐, 100,000 轮迭代) → 64 字符十六进制哈希
```

- 盐：`secrets.token_hex(16)` 生成 32 字符随机值
- 迭代次数：100,000（NIST 推荐最低值）
- 哈希算法：SHA256（内置，无外部依赖）

### 4.3 JWT 实现 (`auth.py`)

自实现 HMAC-SHA256 JWT，三段式 `header.payload.signature`：

```
Header: base64({"alg":"HS256","typ":"JWT"})
Payload: base64({"username":"...","role":"...","iat":...,"exp":...})
Signature: SHA256(header.payload.secret_key)
```

**签证验证逻辑**：收到 token → 用 secret 重新计算签名 → 比对是否一致。任何篡改都会导致签名不匹配。过期时间 7 天。

### 4.4 登录态保持

```
登录 → 后端返回 JWT → 前端存 sessionStorage
 → 全局 fetch 拦截器自动附加 Authorization: Bearer <token>
 → 后端 get_current_user 依赖解析并验证
 → 过期/无效 → 返回 401 → 前端自动退出
```

`window.fetch` 在 `app.js` 被重写（monkey-patch），所有对 `API_BASE` 的请求自动注入 token。

### 4.5 历史记录去重 (`auth.py`)

同一文件连续分析 → 覆盖旧记录而非新增。实现：检查该用户最近一条记录的 `filename` 字段，相同则 `UPDATE` 而非 `INSERT`。

## 5. 数据可视化引擎

### 5.1 图表渲染器 (`frontend/app.js`)

支持 15 种图表类型，统一入口 `renderAnalysisChart(domId, data)`：

| 图表类型 | ECharts 配置 | 适用场景 |
|---------|-------------|---------|
| histogram/bar | `bar` series | 分布、频次 |
| horizontal_bar | `bar` + 翻转 x/y 轴 | 排名、分类对比 |
| grouped_bar | 多 series `bar` | 分组对比 |
| stacked_bar | `bar` + `stack: true` | 交叉分析 |
| line | `line` + `smooth: true` | 时序趋势 |
| scatter | `scatter` | 相关性、PCA、聚类 |
| heatmap | `heatmap` | 相关性矩阵 |
| boxplot | 模拟箱线（bar + scatter min/max） | 分组比较 |
| donut/pie | `pie` + `radius: ["45%","70%"]` | 构成占比 |
| pareto | 双 y 轴 bar + line | 帕累托分析 |
| qq | scatter + 对角线 | 正态性检验 |

每个图表自动挂载 `ResizeObserver` 响应容器尺寸变化。

### 5.2 预测沙盘可视化 (`app.js`)

**贡献瀑布图** (`updateWaterfallChart`)：

水平柱状图，每个特征一根柱子：
- 绿色 = 正向贡献 (coefficient × value > 0)
- 红色 = 负向贡献
- 虚线标注最终预测值 ± 95% 区间

数据流：`sandboxSliderValues[feature] × coefficients[feature]` → 渲染。

**场景对比** (`updateScenarioChart`)：

当前值（蓝色）+ 已保存场景（灰色）并列柱状图，虚线标注预测区间上下限。

**敏感性龙卷风图** (`renderSensitivityChart`)：

计算每个特征从 min 到 max 的影响幅度 = |coefficient × (max - min)|，降序排列。

### 5.3 CountUp 动画 (`app.js`)

`requestAnimationFrame` + easeOutCubic 缓动函数，400ms 从旧值平滑滚动到新值。

currentSchema // 当前数据表结构
currentFilename // 当前工作文件名
profileData // /data-profile 结果缓存

### 6.2 持久化策略

| 数据 | 存储位置 | 生命周期 |
|------|---------|---------|
| LLM 配置 | `sessionStorage` | 关闭标签页清除 |
| 登录令牌 | `sessionStorage` | 同上 |
| 隐私声明 | `localStorage` | 除非勾选"不再提醒" |
| 训练模型 | `sessionStorage` | Tab 4→Tab 5 传递 |
| 分析历史 | 后端 SQLite | 永久，按用户隔离 |

### 6.3 Tab 切换动画 (`app.js`)

使用 CSS `@keyframes` 实现滑动过渡：
- 向后切换：旧面板滑出左，新面板滑入右
- 向前切换：旧面板滑出右，新面板滑入左
- `animationend` 事件监听 → 移除动画 class → 挂 `hidden`

## 7. 流式通信协议 (SSE)

### 7.1 后端实现 (`main.py`)

```python
async def event_generator():
 yield _sse_event("translation", {...}) # 阶段 0+1 结果
 yield _sse_event("plan", {...})
 yield _sse_event("progress", {...}) # 每步进度
 yield _sse_event("result", {...}) # 分析结果
 yield _sse_event("interpretation", {}) # 逐 token 解读
 yield _sse_event("done", {})

return StreamingResponse(
 event_generator(),
 media_type="text/event-stream",
 headers={
 "Cache-Control": "no-cache",
 "Connection": "keep-alive",
 "X-Accel-Buffering": "no", # 禁用 nginx 缓冲
 },
)
```

### 7.2 前端解析 (`app.js`)

使用 `fetch()` + `ReadableStream` 而非 `EventSource`，因为需要 POST 请求。

**行缓冲算法**：
1. 文本追加到 buffer
2. `split("\n")` 分割
3. `lines.pop()` 保留最后不完整行，放回 buffer 等待下一个 chunk
4. 逐行解析：`event:` 设置事件类型，`data:` 触发回调

**为什么不直接用 EventSource**：`EventSource` 只支持 GET，而 `/analyze` 需要 POST 传提问内容和 LLM 配置。

### 7.3 断连处理

前端 `reader.read()` 异常时自动设置 `analysisLoading = false` 恢复 UI。后端 `StreamingResponse` 中断时 FastAPI 自动清理 async generator。

## 8. 安全设计

### 8.1 密码安全

PBKDF2-SHA256 + 随机盐 + 100k 迭代。密码从不以明文存储、传输或日志输出。

### 8.2 API Key 安全

- 前端存 `sessionStorage`，不存 `localStorage`
- 每次请求随 Body 传，后端用完即弃
- 不落盘、不入日志、不存数据库

### 8.3 AI 安全边界

- AI 绝不生成 Python 代码
- AI 只输出 JSON 方案，Python 执行预定义函数
- **双层列名校验**（2026-05-26 强化）：Prompt 层约束 actions 与 column_mappings 列名一致性；执行层通过 registry input_schema 自动识别列参数，与 DataFrame 实际列名比对，不匹配立即拦截
- **优雅降级**：无法分析时 AI 诚实说明原因，不编造结论；列缺失时精确报告哪些列不存在及可用列列表
- LLM 只接收 Schema 元数据 + 前 3 行样本，不接触完整数据

### 8.4 认证鉴权

- 核心端点 (`/upload`, `/analyze`, `/train`) 强制登录
- JWT 签名防篡改，过期自动失效
- 管理端点 (`/users`, `/admin/*`) 额外校验 admin 角色

### 8.5 隐私保护

每次登录弹出隐私声明，承诺数据不外泄。用户可勾选"下次不再提醒"(localStorage 持久化)。

### 8.6 数据清洗安全

清洗后另存为新文件，原始文件保留不修改。所有清洗操作可撤销。

## 9. 系统鲁棒性增强（2026-05-26）

### 9.1 统一错误弹窗系统 (`frontend/app.js`)

所有用户可见的错误统一走 `showErrorModal(title, message, hint?)`：
- 居中模态弹窗 + 半透明遮罩 + 模糊背景
- 点击遮罩或"知道了"按钮关闭
- 替代了原有的 `showToast()`、`alert()`、内联红色文字三种不一致的错误展示

错误来源覆盖：
| 场景 | 触发方式 |
|------|---------|
| 401 登录过期 | fetch 拦截器自动弹出 |
| 上传失败 / 格式错误 / 超 50MB | 前端校验 + 后端 413 |
| 训练失败（非数值列） | `/train` 返回 400 |
| SSE 分析异常 | `event: error` → `showErrorModal` |

### 9.2 文件上传 50 MB 限制 (`main.py` + `app.js`)

- **前端**：`validateFile()` 检查 `file.size > 50 * 1024 * 1024` → 弹窗
- **后端**：`seek(0, 2)` 获取文件大小，超限返回 HTTP 413 + 明确提示

### 9.3 SSE 流异常兜底 (`main.py`)

`/analyze` 的 `event_generator` 在三个关键阶段增加了 SSE `error` 事件推送：

| 阶段 | 异常 | 前端行为 |
|------|------|---------|
| LLM 合并调用 | 网络错误 / API Key 无效 | `event: error` → 弹窗提示检查配置 |
| 列映射 | `apply_column_mappings` 异常 | `event: error` → 弹窗，继续后续分析 |
| AI 解读 | LLM 调用失败 | `event: error` + 内联降级文字 |

### 9.4 JSON 解析多级回退 (`llm_agent.py` `_extract_json`)

```
第 1 级：去除 markdown 代码块 (```json ... ```)
第 2 级：去除 JSON 前后的自由文本
第 3 级：移除尾逗号（LLM 常见错误）
第 4 级：单引号 → 双引号（Python dict 语法兜底）
失败 → ValueError + 前 200 字符上下文
```

### 9.5 训练前预览端点 (`main.py` `/preview-train`)

在正式训练前快速检查数据质量，返回：
- 总行数 / 有效样本数 / Y 缺失丢弃率
- 每个特征的 NaN 比例
- 非数值列的提前拦截

前端先调预览再训练，用户看到预期后再发起实际拟合。

### 9.6 会话上下文 LRU 淘汰 (`main.py`)

`_session_context` 通过 `_set_context()` 写入，超过 20 个文件自动淘汰最旧记录，防止内存无限增长。

### 9.7 高基数特征检测 (`ml_trainer.py`)

训练前检查 `nunique / n_samples > 0.5` 的特征，标记为"可能是 ID 列"（如 CustomerID），在结果 `warnings` 字段返回，前端训练日志展示警告。
